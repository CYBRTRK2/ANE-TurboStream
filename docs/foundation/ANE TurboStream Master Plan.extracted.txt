PHASE 0  llama.cpp Foundation — Stock Baseline
Target: 7–10 tok/s  —  The reference everything is measured against
0A1628
Goal
Build the anemll Flash-MoE llama.cpp fork from source on the M4 Air. Run Qwen3.5-35B-A3B IQ2_M in stock mode (all weights in RAM, no slot-bank, pure GPU). Establish the clean benchmarked baseline. Every future phase improvement is measured relative to this number.
Why llama.cpp and Not the Old infer.m
The previous project used a custom Objective-C/Metal runtime (infer.m). That work is archived. The architectural decision to switch to llama.cpp is based on three things: (1) the Flash-MoE slot-bank for SSD streaming is already production-grade in the anemll fork and would take weeks to rebuild in infer.m from scratch; (2) llama.cpp's ggml_backend system is the correct foundation for adding an ANE backend later via ggml-coreml or ggml-ane; (3) GGUF model support means Qwen3.5, Kimi K2, Cascade-2, and any future model works without custom loaders. The infer.m benchmark evidence remains in the project folder as historical reference for timing measurements.

PHASE 1  SSD Streaming — Flash-MoE Slot-Bank
Target: 15–25 tok/s
The primary performance lever on 16 GB
Goal
Activate the Flash-MoE slot-bank. Routed expert weights move to SSD. Only N=32 resident expert slots stay in GPU memory per MoE layer. Dense weights (attention, norms, embedding, shared experts, LM-head) stay in GPU memory via -ngl 99. The slot-bank hit path (expert already resident) is fast. The miss path (pread from SSD + evict victim slot) is the I/O cost to minimize.
Why This Is the Primary Lever
Qwen3.5-35B at IQ2_M: routed expert weights make up ~90% of total model size. In stock mode, all 256 experts per layer are loaded into the 16 GB unified memory pool simultaneously. The slot-bank keeps only 32 of 256 experts resident (~1.1 GB overhead) and streams the rest from SSD on demand. This frees ~6 GB of RAM that was previously occupied by cold experts, reducing memory pressure on the GPU and improving throughput.
Step 1.1 — Sidecar Extraction
Extract routed expert weights from the GGUF into per-layer binary sidecar files on disk. These files never load into RAM. The slot-bank pread()s from them on miss.
# Read the sidecar tool documentation first
cat references/anemll-flash-llama.cpp-master/tools/flashmoe-sidecar/README.md
# Extract sidecar
python tools/flashmoe-sidecar/extract_sidecar.py \
--gguf /path/to/Qwen3.5-35B-A3B-IQ2_M.gguf \
--output /path/to/qwen35b_iq2m_sidecar/
The sidecar extraction produces one binary file per expert per layer. Total sidecar size approximately equals the routed expert portion of the GGUF (~6-7 GB). Store on the internal NVMe for maximum pread() throughput.

Step 1.4 — Oracle Ceiling Measurement
The oracle-all-hit mode simulates perfect cache behavior: every expert requested is already in the slot-bank, zero SSD reads. This gives you the hard speed ceiling for the current slot-bank size. The gap between your slot-bank result and the oracle ceiling tells you how much is lost to cache misses.

PHASE 2  TurboQuant KV Cache Compression
Target: 18–28 tok/s
Smaller KV = faster attention = longer context
Goal
Compress the KV cache approximately 3.5x using TurboQuant (arXiv 2504.19874, Google DeepMind). Smaller KV cache reduces attention memory pressure, enables longer context windows within 16 GB, and indirectly reduces cmd1_wait (the GPU wait after the attention command buffer). This is the highest-leverage algorithmic change available without touching the model weights.
The Algorithm — Credit: Google DeepMind, arXiv 2504.19874
Before implementing anything, read the paper. The SwiftLM reference implementation is at references/SwiftLM-main/Sources/SwiftLM/Server.swift and references/SwiftLM-main/Sources/MLXInferenceCore/InferenceEngine.swift. Port the math to MSL (Metal Shading Language). Do not copy Swift code.
K-cache compression: 4.25 bits/dim
Step 1: Extract L2 norm, normalize: x_hat = x / ||x||
Step 2: Apply Fast Walsh-Hadamard Transform (WHT) in-place. Distributes outliers evenly across dimensions before quantization.
Step 3: Quantize each coordinate using 3-bit non-linear Lloyd-Max centroids (8 centroids, optimized for Gaussian distribution post-WHT rotation). Read paper Section 3.2 for exact centroid values.
Step 4: Compute residual: r = x_hat - dequant(quantized)
Step 5: Project residual via random Johnson-Lindenstrauss (QJL) matrix, store 1-bit sign. QJL acts as inner-product regularizer. Prevents centroid resolution loss from degrading attention dot-product scoring.
V-cache compression: 3.125 bits/dim
Steps 1-3 identical to K-cache.
Step 4: Stop. No QJL for V-cache. V-cache is used for weighted accumulation, not inner-product scoring. QJL provides zero benefit and wastes bits.
Do not use approximate centroid values from memory. Read Section 3.2 of arXiv 2504.19874 for the exact Lloyd-Max centroid table before writing the shader.

PHASE 3  GPU Optimizations — Reduce Waits and LM-Head
Target: 22–32 tok/s
Squeeze remaining GPU performance before ANE
Goal
Attack the remaining per-token GPU wait time. After Phase 2, the dominant bottlenecks will be cmd1_wait (GPU wait after attention command buffer), cmd2_wait (GPU wait after MoE command buffer), and lm_head (full vocabulary projection at each decode step). This phase squeezes them before adding the complexity of ANE dispatch.
Step 3.1 — Instrument the Timing
Add per-phase timing instrumentation to the llama.cpp Metal backend, mirroring what was done in infer.m. Split cmd1_wait from cmd1_flush and cmd2_wait from cmd2_flush. This is how you find the real bottleneck rather than guessing.
// In ggml-metal.m, around waitUntilCompleted calls:
NSDate* t0 = [NSDate date];
[cmd_buf_1 waitUntilCompleted];
double cmd1_wait_ms = -[t0 timeIntervalSinceNow] * 1000.0;
Step 3.2 — cmd1_wait Reduction
cmd1_wait is the CPU blocking on the attention command buffer. The question: does top-K expert routing start after full cmd1 completion, or is there overlap potential? Read the ggml-metal.m source and find every waitUntilCompleted call. Map the data dependency chain. If gate logit readback is inside cmd1's render pass, the wait is unavoidable. If not, there may be overlap.
What to look for in ggml-metal.m
Where is the gate projection matmul encoded?
Where does the CPU read back the gate logits for top-K selection?
Is the readback inside cmd1 or cmd2?
Is there a dependency between cmd1 output and cmd2 input that forces the full wait?
Do not propose moving waitUntilCompleted without tracing the full data dependency chain. Moving a wait that is actually required will produce incorrect output. The quality gate will catch it if it does.
Step 3.3 — LM-Head Optimization
The LM-head at decode time is a matrix-vector multiply: hidden_dim (7168) by vocab (152K). This is memory-bandwidth-bound at batch size 1. Two approaches worth investigating:
Chunked parallel dispatch: split the vocab projection across multiple compute encoders dispatched simultaneously to different GPU cores. Check if ggml-metal.m already does this.
Top-P early exit: compute logits in sorted chunks, stop once the cumulative probability mass exceeds the top-P threshold. Avoids computing the full 152K logits when only the top-1000 matter.
The two previously rejected LM-head approaches (--gguf-lm-head and --nax) are documented in the execution history. Do not re-propose them. Read the rejected.txt files in the old runs/backend/ directories before designing any new LM-head approach.

PHASE 4  ANE Integration — Activate the Neural Engine
Target: 30–40 tok/s
The layer no existing LLM engine uses
Goal
Route eligible FP16 static projection operations off the GPU and onto the ANE. The ANE delivers 38 TOPS at 6.6 TFLOPS/watt. Every other LLM engine on Apple Silicon (llama.cpp stock, MLX, Ollama) shows ANE at 0% during inference. Phase 4 changes that. The visual proof is a mactop screenshot showing ANE power draw > 0 W during generation.
Why Phase 4 Comes After Phase 1-3, Not Before
Three reasons this order is non-negotiable:
IQ2_M weight constraint: the ANE cannot execute 2-bit weight operations natively. CPU dequantization to FP16 before ANE dispatch costs more than the ANE compute gain. This is why the original ANE probe ran slower. The Phase 1 slot-bank is what enables FP16 dense layers while keeping expert weights at IQ2_M on SSD.
Memory headroom: ANE staging buffers need free memory. Phase 1+2 together free approximately 4 GB (slot-bank frees expert RAM, TurboQuant compresses KV). Without this headroom, ANE buffers would OOM.
Clean baseline: Phase 3 instruments every GPU phase precisely. Without that instrumentation, you cannot measure whether ANE dispatch is actually helping or hurting.
The ANE Dispatch Rule
ANE dispatch is only worth attempting when ALL THREE conditions are true. Check all three before proposing any ANE offload.

Implementation Path: CoreML Backend for llama.cpp
There are two paths to ANE dispatch. Start with Path A. Move to Path B only if Path A proves insufficient.
Path A: CoreML (ggml-coreml.m) — Recommended First
Compile the eligible ops (QKV, O, shared experts) as a CoreML MLModel at model load time.
At inference, dispatch via MLModel API. CoreML automatically routes eligible FP16 matmuls to the ANE.
No private API access. Works with App Store if ever needed.
Limitation: you do not control which hardware unit CoreML uses. It decides based on op size.
Path B: Private ANE APIs (_ANEClient) — If Path A Is Insufficient
Use the reverse-engineered private APIs from maderix/ane (referenced in docs/ane_lane2_notes.md).
Compile MIL kernels via _ANECompiler, dispatch via _ANEClient + IOSurface buffers.
Full control. ANE utilization visible in mactop as hard evidence.
Risk: private APIs may break on macOS updates.
Path B is what the maderix/ane project documented. The ane_lane2_notes.md file in the project explains what the original probe attempted. The difference in Phase 4: we are dispatching FP16 dense-layer ops (post slot-bank), not IQ2_M ops. That is what was wrong with the original probe.

PHASE 5  EvoDistill — Gradient-Free On-Device Fine-Tuning
Target: Better model quality on same hardware
No GPU cluster. No backward pass. ANE forward passes only.
Goal
Fine-tune the model to produce better outputs on code generation tasks, running entirely on the M4 Air using the ANE for forward passes. Zero backward passes. Zero GPU cluster. Zero external annotation. Uses Apple's SSD method (arXiv 2604.01193) as the training signal and EGGROLL (@rustane_dev, March 2026) as the gradient-free optimizer.
Phase 5 is independent of Phases 1-4 for the training pipeline. But the final fine-tuned model must be deployable on the Phase 1-4 inference stack. Do not begin Phase 5 until Phase 1 (slot-bank) is working and quality-gated. The fine-tuned model needs a working inference engine to serve it.

The EvoDistill Fusion
EvoDistill replaces SSD's Stage 2 (the only backward pass in the original pipeline) with EGGROLL's ES optimizer. The substitution is valid because ES minimizes any loss function using only forward pass evaluations. Result: the entire pipeline runs via forward passes. The ANE handles all of them.

E2: ES Optimizer (Stage 2, EGGROLL algorithm, credit: @rustane_dev)
def es_step(theta, corpus, model, sigma=0.01, P=16, alpha=0.01):
grad_est = zeros_like(theta)
for k in range(P // 2):
eps = randn_like(theta) * sigma
F_pos = -cross_entropy(inject_lora(model, theta + eps), batch(corpus))
F_neg = -cross_entropy(inject_lora(model, theta - eps), batch(corpus))
grad_est += (F_pos - F_neg) * eps
return theta + (alpha / (P * sigma)) * grad_est
Target layers: deeper 24 of 48 (EGGROLL's key empirical finding). Run for S=200 steps. Log val_loss every 10 steps.

Phase 5 Rejection Criteria
val_loss at step 200 does not improve over step 0 by at least 5%: sigma or layer selection is wrong. Fix before continuing.
T* == T_train exactly: something is wrong with val set or calibration sweep. Investigate.
Round K val_loss >= round K-1: stop the loop at K-1. Do not continue.
Final LiveCodeBench V6 pass@1 does not exceed frozen baseline by at least 3 pp: EvoDistill does not work on this configuration. Report honestly. Do not promote.
Quality gate fails at any checkpoint: stop immediately.
